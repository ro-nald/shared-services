# IAM Environment

Manages Terraform deployer roles and governance controls for all environments in this
repository. This is **Stage 1** — `environments/core/` (Stage 0) must be applied first.

## What it manages

| Resource | Name pattern | Purpose |
|---|---|---|
| IAM roles | `terraform-deployer-*` | Assumed by Terraform to deploy environments |
| IAM policies | `terraform-deployer-*-policy` | Scoped permissions per role |
| Permission boundary | `team-deployer-boundary` | Hard ceiling on all team deployer roles |

## Prerequisites

- Terraform ≥ 1.10
- Stage 0 applied — S3 state bucket available from `environments/core/` outputs
- `backend.hcl` generated by the bootstrap CLI (`uv run scripts/bootstrap.py migrate-state iam`)

## Applying (Stage 1)

In CI this environment is applied automatically by the `apply-iam` workflow job (with
a required reviewer gate) after a PR merges to `main`.

For local runs after the bootstrap CLI has been used:

```bash
cd terraform/environments/iam
terraform init -backend-config=backend.hcl
terraform apply
```

After a successful apply, note the deployer role ARN — you will need it for Stage 2:

```bash
terraform output terraform_deployer_dev_role_arn
```

## File structure

```text
environments/iam/
├── main.tf                   # Provider config and bootstrap comments
├── variables.tf              # Input variables
├── terraform.tfvars          # Variable values
├── outputs.tf                # Role ARNs and next-steps guidance
├── permission-sets.tf        # Platform-owned reusable permission building blocks
├── platform-bootstrap.tf     # IAM role for re-running apply-core without admin credentials
├── roles.tf                  # Platform deployer role + permission boundary policy
└── teams/
    ├── .gitkeep
    ├── team-example.tf.example   # Copy this to add a new team role
    └── team-<slug>.tf            # One file per team (added via PR)
```

## Adding a team role

Teams request their own Terraform deployer role by opening a pull request to this
repository. The PR is the formal record of the permission request.

### Steps

1. **Copy the example file** into `teams/`:

   ```bash
   cp teams/team-example.tf.example teams/team-<your-slug>.tf
   ```

2. **Replace `example`** with your team slug throughout the file (role name, resource
   names, tags).

3. **Select permission sets** from `permission-sets.tf`. Uncomment the lines for the
   AWS services your team's Terraform code manages:

   | Permission set data source | Services covered |
   |---|---|
   | `pset_ecr_manage` | ECR repositories and images |
   | `pset_s3_manage` | S3 buckets and objects |
   | `pset_ec2_manage` | EC2 instances, AMIs, security groups |
   | `pset_vpc_manage` | VPCs, subnets, route tables, internet gateways |
   | `pset_iam_service_roles` | IAM roles and policies for services (not user management) |
   | `pset_sts_caller_identity` | Read current AWS identity (needed by most deployments) |
   | `pset_s3_team_state` | Scoped read/write to the team's own state prefix in the shared S3 bucket |
   | `pset_ssm_shared_read` | Read-only access to shared-services SSM outputs under the platform namespace |

   Do **not** add raw IAM actions directly to your team file. If you need a permission
   that does not exist in any permission set, open a separate PR to `permission-sets.tf`
   — that file requires platform team approval only.

4. **Open a pull request**. Required reviewers are set in `.github/CODEOWNERS`:
   - Your team lead
   - The platform team

5. After merge, the CI pipeline runs `terraform apply`. Once complete, the role ARN
   appears in the `terraform output` and is tagged `Purpose=terraform-deployer` for
   discovery by the role-picker CLI.

### Example team file

```hcl
# teams/team-payments.tf

data "aws_iam_policy_document" "team_payments_permissions" {
  source_policy_documents = [
    data.aws_iam_policy_document.pset_ec2_manage.json,
    data.aws_iam_policy_document.pset_s3_manage.json,
    data.aws_iam_policy_document.pset_vpc_manage.json,
    data.aws_iam_policy_document.pset_sts_caller_identity.json,
  ]
}

resource "aws_iam_policy" "team_payments_deployer" {
  name   = "terraform-deployer-team-payments-policy"
  policy = data.aws_iam_policy_document.team_payments_permissions.json
  tags   = merge(var.tags, { Team = "payments", Environment = "dev" })
}

resource "aws_iam_role" "team_payments_deployer" {
  name                 = "terraform-deployer-team-payments"
  assume_role_policy   = data.aws_iam_policy_document.terraform_trust.json
  permissions_boundary = aws_iam_policy.team_deployer_boundary.arn
  max_session_duration = 3600
  tags = merge(var.tags, {
    Purpose     = "terraform-deployer"
    Team        = "payments"
    Environment = "dev"
  })
}

resource "aws_iam_role_policy_attachment" "team_payments_deployer" {
  role       = aws_iam_role.team_payments_deployer.name
  policy_arn = aws_iam_policy.team_payments_deployer.arn
}
```

## Outputs reference

| Output | Description |
|---|---|
| `aws_account_id` | AWS account ID |
| `terraform_deployer_dev_role_arn` | ARN of the platform deployer role for the dev environment |
| `platform_bootstrap_role_arn` | ARN of the IAM role for re-running `apply-core` without admin credentials |
| `next_steps` | Guidance on how to use the role ARN in Stage 2 |

## Discovering roles (role-picker CLI)

A separate CLI tool (maintained in its own repo) discovers all deployer roles tagged
`Purpose=terraform-deployer` and presents an interactive selection. It calls:

```bash
# Tag-based discovery (preferred)
aws resourcegroupstaggingapi get-resources \
  --resource-type-filters iam:role \
  --tag-filters Key=Purpose,Values=terraform-deployer

# Name-prefix fallback
aws iam list-roles \
  --query "Roles[?starts_with(RoleName, 'terraform-deployer-')].[RoleName,Arn]"
```
